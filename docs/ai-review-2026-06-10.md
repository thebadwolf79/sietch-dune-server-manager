## A. Consensus: improve the *existing* code first (these de-risk everything else)

Ordered by leverage. The first three are "foundation" — every planned feature gets easier,
safer, and testable once they land.

### A1. Command-execution abstraction + rich result type — **HIGH (unanimous)**
Replace `kubectl/mod.rs::require_ok` (errors on *any* non-zero exit) with a `CommandRunner`
trait returning a structured `CommandOutput { exit_code, stdout, stderr }`, plus a
`MockCommandRunner` for tests. Success is then interpreted **per-command in business logic**
(e.g. `update()` accepts exit 0 *or* 1) instead of a generic unforgiving helper.
- *Why:* kills the scattered "tolerate exit 1" hacks (#24), and makes the whole SSH/kubectl
  layer unit-testable without a VM. (Gemini #1, Grok #1, ChatGPT #1, Perplexity §2.)

### A2. Verify *state*, not exit codes (k8s reconciliation mindset) — **HIGH**
After running vendor `battlegroup` commands, confirm the desired state out-of-band: CR status,
`kubectl get … -o json`, version comparison (`battlegroupVersion` vs `liveBattlegroupVersion`),
or symlink/readlink checks — rather than trusting the script's exit code. This is exactly how
upstream fixed the sibling `restart` bug (#20). (Perplexity §2, Grok, ChatGPT #1.)

### A3. Derive status from the Director/BGD health endpoints (or 1.4.5.0's status), not raw pod phase — **HIGH**
The app infers status from k8s pod phase + DB state; the official tool reads the Director (BGD).
Result: wrong uptime (BG vs pod), "Running" during Startup, "Initializing/Running" instead of
"Healthy" (**#21**), and no grace-period players (**#14**). Use `kubectl … -o json` / the `kube`
crate for **structured** reads, and prefer the Director endpoints. **1.4.5.0's new "proper
Start/Stop polling" likely exposes this — adopting it may fix #21 for free.** (All four models;
Gemini and Grok both flag this as the highest-value pivot.)

### A4. Explicit lifecycle state machine + wire VM power-on — **HIGH**
Model the whole lifecycle as states (`VmOff → VmStarting → VmRunning → ServerStarting →
ServerHealthy → ServerDegraded → …`) and **gate every UI action on the current state**.
`start_vm()/stop_vm()` already exist in core but aren't surfaced — wire them in and present a
"Start Server" action when the VM is off instead of hard-failing. (Gemini #3, Grok #4,
ChatGPT #2.) **All models say reorder this to #1 in our plan** — it's the foundation the
availability-gating and freeze fixes build on.

### A5. Background tasks + per-call timeouts + cancellation + gated polling — **HIGH**
Run SSH/kubectl on `tauri::async_runtime::spawn` (or `spawn_blocking` + outer timeout); never
block the UI thread. Use short (1–3 s) timeouts for UI-driven polls with exponential backoff,
a `CancellationToken` per panel, and **gate all polling on "VM up + SSH reachable"** so a dead
endpoint never freezes the UI. (Fixes **#25**'s freeze and powers **#22**.) (Perplexity §1, Grok #5.)

### A6. Persist settings in the Rust core (single source of truth) — **MEDIUM-HIGH**
A `Settings` struct loaded once, held in a `RwLock`, persisted to a config file, exposed via
`get_settings`/`set_settings` Tauri commands. The frontend must initialize from `get_settings`,
never hardcoded defaults — that's the root of the auto-update toggle reverting (**#25**).
(Gemini #5, Perplexity §1c.)

### A7. Rich typed errors + auto-surface failure context — **MEDIUM-HIGH**
Adopt `thiserror` domain errors. On a backup/dump failure, **automatically run `kubectl logs` /
`describe` and package the output into the error** shown in-UI — instead of "run kubectl
describe yourself." Distinguish OOM (exit 137) from the stray-table permission error (exit 1).
(Fixes **#7**'s opaqueness.) (Gemini #4, ChatGPT #5, Grok #8.)

### A8. pg_dump hardening + SSH efficiency — **MEDIUM**
- *Stray-table exit 1:* give the dump a role with `SELECT`/`USAGE` on all relevant schemas (or
  scope the dump by schema/owner) so a stray non-`dune` table can't abort it. (Perplexity §4b.)
- *OOM (#23):* raise the dump pod's memory request/limit, use low-level streaming compression,
  reduce `pg_dump` parallelism. (Perplexity §4a, Grok #6.)
- *Efficiency:* reuse/multiplex SSH connections, batch round-trips, small TTL cache for
  idempotent reads — the status panel currently does many sequential SSH sessions. (Grok #3/#7.)

### A9. Move status/state logic out of the TS frontend into Rust — **MEDIUM**
`utils/remote-server.ts` holds business logic that belongs in the core (single source of truth,
testable, reusable by a future CLI). Frontend renders state + calls declarative commands.
(Gemini #2.)

## B. Verdict on our planned fork work

**Reorder to:** (1) VM power-on + state machine → (2) status pivot to Director/1.4.5.0 →
(3) pg_dump hardening *with* rich-error surfacing → (4) currency grant + filterable picker →
(5) char admin + DR backup → (6) guardrails → (7) #23 OOM support.

- ✅ **Right:** pg_dump hardening, VM power-on, guardrails, supporting #23 — all endorsed.
- 🔁 **Reorder:** VM power-on/state-machine first (it's the foundation, per all models); pivot
  status to the new endpoints *early* since it likely closes #21 cheaply.
- ⚠️ **1.4.5.0 redundancy:** do **not** re-implement `change-vm-ip`, `change-battlegroup-ip`,
  per-map "Minimum Servers", or `enable-experimental-swap`. The vendor owns these now —
  **invoke / read / display** them (wrap the new subcommands, show current values, offer "open
  config"). (Grok, Gemini.)
- ⚖️ **DR backup (§3.7) — split opinion:** Gemini calls it the killer feature; Grok warns it's
  over-scoped for a fork and risks forking the data model, and recommends keeping restore
  minimal (throwaway-DB only). **Resolution:** build the *identity-capture* half (genuinely
  unique value — nothing else solves reinstall-orphaning) but keep the restore path minimal and
  guarded, and **do the binding-identifier investigation first** (it de-risks the whole thing).
  - *Experiment design (to find the binding identifier safely):* on a throwaway box, do a
    controlled reinstall that **reuses `WORLD_UNIQUE_NAME` + `FLS_TOKEN`** but lets RMQ/DB
    secrets regenerate, then `battlegroup import` a dump and see if the character loads. If it
    loads → FLS identity is sufficient and the secrets don't bind the character; if not →
    capture/replay the secrets too. Confirm the k8s CRD `metadata.uid` is *not* the binding key.
- ➕ **Missing (add to plan):** SSH connection reuse; a **self-host connectivity/health panel**
  (VM reachable? k3s API up? required ports forwarded? RAM vs recommended?) — directly targets
  the community pain below (Perplexity §6); and the command-runner/state-machine refactors as
  explicit line items, not incidental.

## D. Tools & skills to build our fork from source for testing

From [`building-from-source.md`](./building-from-source.md): **Rust stable, Node.js 22, npm,
Git, Tauri CLI**; Windows needs the **WebView2** runtime.

**Local build/test loop (Windows — our platform):**
```powershell
cd app
npm ci
npm run tauri -- dev                 # hot-reload dev build for iterating on fixes
npm run tauri -- build --no-bundle   # local production exe → target/release (for real testing)
cargo check --workspace; cargo test --workspace
```

**Tooling/skills we already have that accelerate this:**
- **`context7` MCP** (`resolve-library-id` + `query-docs`) — pull current **Tauri v2**, **kube-rs**,
  `thiserror`, `tokio` docs while implementing A1–A7 (avoids stale-API guesses).
- **Agent `isolation: "worktree"` / `EnterWorktree`** — build & test fork changes in an isolated
  git worktree so `main` stays clean; ideal for the command-runner refactor.
- **`make-plan` + `do` skills** — phase the implementation and execute with subagents.
- **`code-review`, `simplify`, `verify` skills** — review our diff, then `verify` actually runs the
  built app to confirm a fix works (not just compiles).
- **`gateguard` skill** — fact-forcing gate before edits (quality guardrail on risky changes).
- **`docker-patterns` skill + Docker** — only if we pursue the **Linux AppImage** path
  (Mhynlo's closed PR used an Ubuntu 26.04 container); blocked upstream by the PowerShell-only
  backend, so Windows-first is the pragmatic test target.
- **`github` MCP** — fork already exists at `thebadwolf79/dune-dedicated-server-manager`;
  use it to branch, `push_files`, and open the PRs above.
- **Testing aid:** the `MockCommandRunner` from A1 lets us unit-test exit-code/status logic with
  no VM — pair with `cargo nextest` for fast runs.
